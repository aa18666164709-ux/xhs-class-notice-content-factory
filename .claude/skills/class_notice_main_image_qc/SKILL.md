# Class Notice Main Image QC

## 触发语

- 质检6张主图
- 运行主图质检
- 生成主图拼图
- 检查本轮主图
- 运行主图归档并上传GitHub

## 核心原则

主图决定点击率，所以每轮 6 套笔记只深度复核 6 张主图。

副图不做视觉质检，只要求发布包里文件存在、命名正确即可。

## 质检对象

固定扫描根目录：

`E:\优逸店资料截图\班级通知\笔记素材\生成成品图片`

每轮优先扫描本轮 6 个发布包文件夹。若无法确定“本轮”是哪 6 个文件夹，先列出候选发布包并请用户确认，不要擅自挑选。

## 标准使用流程

默认流程如下：

1. 6 个做图窗口各做 1 张主图。
2. 每张主图保存到各自发布包文件夹，文件名必须是 `主图.png`。
3. 用户回到本地知识库 / Codex 窗口说：`运行主图归档并上传GitHub`。
4. Codex 扫描本轮 6 个发布包文件夹。
5. Codex 确认 6 张 `主图.png` 都齐了。
6. Codex 生成 6 张主图拼图预览。
7. Codex 把压缩后的拼图上传到 GitHub 临时目录。
8. 用户让 ChatGPT 看 GitHub 临时图片，或让 Codex 把拼图路径发给用户。
9. 用户和 ChatGPT 只复核主图。

如果 6 张主图未齐，必须停止并列出缺失的发布包，不生成拼图，不上传 GitHub。

## 每个发布包读取内容

重点读取：

- `主图.png`
- `发布文案.md`
- `info.json`

副图只做存在性检查：

- `副图1.png`
- `副图2.png`
- `副图3.png`
- `副图4.png`
- `副图5.png`

严禁对副图做视觉质量分析。副图不判断好不好看，不消耗额外额度做深度分析。只要副图存在、命名正确，就默认通过。

## 主图质检标准

每套只检查 `主图.png`，重点判断：

1. 主标题是否足够大
2. 主标题是否是具体老师场景
3. 是否突出“老师不用跑教室叫人”
4. 是否出现办公室老师 + 教室一体机/大屏
5. 屏幕文案是否大、短、清楚
6. 是否跑偏成家长通知、家校沟通、回执统计
7. 是否像复杂软件后台
8. 是否广告感过重
9. 是否值得发给 ChatGPT 做最终判断

## 判断结果

每张主图只给三类结论：

- 通过：主标题清楚、场景具体、屏幕文案可读，可以进入发布。
- 需 ChatGPT 复核：主图方向可能能用，但存在标题大小、场景理解、广告感或屏幕文案可读性疑问。
- 建议重做：主图跑偏、主标题不清、屏幕文案看不清、像复杂后台、像家长通知或点击诱因不足。

## 生成 6 张主图拼图预览

把本轮 6 个发布包里的 `主图.png` 拼成一张总览图。

要求：

- 只拼主图
- 不拼副图
- 每张主图下面标注主题或文件夹名
- 文件名：`YYYYMMDD_班级通知_6张主图拼图预览.png`
- 保存到：`18-运行日志/主图质检/`

同时生成一份压缩后的 JPG 给 ChatGPT 临时查看：

- 文件名：`latest_6张主图拼图预览.jpg`
- 保存到：`90-临时给ChatGPT看的图片/latest_6张主图拼图预览.jpg`
- 新一轮生成时可以覆盖旧文件

GitHub 临时目录只放这张压缩拼图，供 ChatGPT 快速看主图质量。

不要同步 6 张主图原图。不要同步副图。

## GitHub 同步边界

运行质检后，只同步：

- 压缩后的主图拼图预览 JPG：`90-临时给ChatGPT看的图片/latest_6张主图拼图预览.jpg`
- 主图质检报告

不要同步 6 张主图原图。不要同步全部副图。不要把完整图片库上传 GitHub。

ChatGPT 复核完成后，提示用户：

“ChatGPT 复核完成后，请执行：清理GitHub临时图片。”

## 完成后输出

完成一次主图质检后，输出：

- 主图拼图本地路径
- 主图拼图 GitHub 路径
- 主图质检报告路径
- 哪些主图通过
- 哪些主图需要 ChatGPT 复核
- 哪些主图建议重做
- 提醒：ChatGPT 复核完成后，请执行“清理GitHub临时图片”。
