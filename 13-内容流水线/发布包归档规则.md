# 发布包归档规则

## 固定目标路径

所有最终要发布的小红书图文笔记，都必须整理到：

E:\优逸店资料截图\班级通知\笔记素材\生成成品图片\

每一套笔记单独一个文件夹。

不要让发布助手直接读取：

C:\Users\Administrator\.codex\generated_images\

`.codex/generated_images` 只作为做图窗口的临时生成目录，最终发布包必须复制到 E 盘标准目录。

## 文件夹命名规则

格式：

YYYYMMDD_主题关键词_全新生成

示例：

20260518_临时叫学生不用跑教室_全新生成

## 固定文件结构

## 发布文案.md 生成规则

`发布文案.md` 给人工检查使用，必须保留真实 Markdown 源码换行。

正文段落之间必须空一行，也就是源码里真实存在两个换行。

禁止：

- 把正文写成一整行。
- 只依赖浏览器自动换行。
- 用浏览器预览效果代替源文件排版检查。
- 只生成 Markdown，不生成 info.json。

推荐格式：

```markdown
# 方案：主题名称

## 正文标题
这里是标题

## 正文内容
第一段正文。

第二段正文。

第三段正文。

## 标签
#班主任 #老师日常 #教师工具

## 首评
这里是首评

## 验收重点
这里写发布前需要检查的重点
```

## info.json 生成规则

发布助手优先读取 `info.json`，不再依赖从 Markdown 浏览器预览复制正文。

每个发布包必须生成 `info.json`，字段包括：

- `title`：小红书标题
- `content`：正文纯文本
- `tags`：标签数组，不带 `#`
- `comment`：首评
- `product.id`：商品 ID，未知时留空
- `product.name`：商品名称，默认 `班级通知`
- `scheduleAt`：定时发布时间，未知时留空

`content` 字段的段落换行必须使用 `\n\n`。

严禁把 `content` 写成一整行。  
严禁只生成 Markdown，不生成 `info.json`。  
严禁让浏览器预览效果代替真实文本换行检查。

标准格式：

```json
{
  "title": "笔记标题",
  "content": "第一段正文。\n\n第二段正文。\n\n第三段正文。",
  "tags": ["班主任", "老师日常", "教师工具"],
  "comment": "这里是首评",
  "product": {
    "id": "",
    "name": "班级通知"
  },
  "scheduleAt": ""
}
```

## 发布文案.md 与 info.json 一致性

`发布文案.md` 里的人眼可读内容，必须和 `info.json` 中的结构化字段一致：

- `## 正文标题` 对应 `info.json.title`
- `## 正文内容` 对应 `info.json.content`
- `## 标签` 对应 `info.json.tags`
- `## 首评` 对应 `info.json.comment`

标签规则：

- `发布文案.md` 中标签可以带 `#`。
- `info.json.tags` 中标签必须不带 `#`。
- `info.json.tags` 必须是数组，不允许写成一个字符串。

## 归档流程

做图窗口生成图片后，图片可能会先出现在：

C:\Users\Administrator\.codex\generated_images\

归档 Skill 必须执行：

1. 判断最终确认的 6 张图顺序。
2. 复制图片到 E 盘标准发布目录。
3. 重命名为固定文件名。
4. 生成 `发布文案.md`。
5. 生成 `info.json`。
6. 校验图片、Markdown、JSON 是否完整。
7. 校验 `info.json.content` 是否包含真实段落换行 `\n\n`。

如果无法判断图片顺序，必须停止并输出所有图片文件名和缩略说明，不要乱命名。

## 发布助手读取规则

发布助手默认读取：

1. 主图.png
2. 副图1.png
3. 副图2.png
4. 副图3.png

`发布文案.md` 只作为人工检查和兜底备份。
