# 项目状态总览

## 系统最高目标

本系统的一切流程、规则、Skill、数据分析、爆款迁移、笔记生成、做图交接、发布包归档和复盘进化，都必须围绕一个商业目标：

提高班级通知产品在小红书的流量、点击、进店、成交和复购可能性。

不以生成数量为核心，不以知识库复杂度为核心，不以自动化炫技为核心。

判断系统是否有效，只看这些指标是否变好：

- 笔记阅读/播放是否提升
- 商品点击是否提升
- 进店人数是否提升
- 订单数是否提升
- 成交金额是否提升
- 高转化内容方向是否越来越清楚
- 低效方向是否被及时停止
- 下一轮内容是否比上一轮更接近成交

## 决策原则

1. 数据优先：有店铺数据时，优先根据数据决策，不凭感觉生成内容。
2. 订单优先：高播放但无点击、无订单的方向不能盲目复刻。
3. 转化优先：低播放但有点击、有订单的方向要优先优化标题和主图继续测。
4. 场景优先：班级通知内容必须围绕老师真实场景，不写抽象软件功能。
5. 复盘优先：每次发布后必须回看数据，把有效规律写回知识库。
6. 简化优先：不能为了系统复杂而复杂，任何新功能都必须服务于数据提升和订单增长。

## GitHub raw 换行校验标准

GitHub raw 默认使用 LF 换行，即 `\n`，不一定是 Windows 的 CRLF `\r\n`。

校验线上 raw 文件行数时，必须使用：

```powershell
$content = (Invoke-WebRequest -Uri "raw地址" -UseBasicParsing).Content
($content -split "`n").Count
```

不要使用：

```powershell
$content -split "`r`n"
```

如果用 `\r\n` 判断，可能会误判为 1 行。

以后判断 GitHub 线上文件是否有真实换行，以以下结果为准：

- LF_COUNT 大于 20：通过
- `content -split "\n"` 行数正常：通过
- GitHub 页面能正常显示 Markdown：通过

不要再仅凭某些网页抽取工具显示“一行”就判断失败。

当前已验证：

- `.gitignore`：GitHub raw 按 `\n` 分割为 38 行
- `PROJECT_STATUS.md`：GitHub raw 按 `\n` 分割为 103 行
